# KIRT Planning Output Comparison: PreFlight vs Forge SDLC

> Comparison of planning artifacts generated by two different systems from the same spec seed (`kirt-spec-seed.md`). Date: 2026-03-24.

---

## 1. Version/Roadmap Coverage

**Spec Seed defines:**
- V1: SAF Phases 1-2 (MVP)
- V2: SAF Phase 3 (Discovery & Scoring)
- V3: SAF Phase 4 (Client Deliverables)
- V4+: SAF Phases 5-6 (Enablement, Continuous Threads, Platform Maturity)

**PreFlight (ideation-lab):**
- HANDOFF.md explicitly lists V1, V2, V3, V4+ with detailed capability breakdowns per version. The deferred capabilities are organized by version (V2, V3, V4+) matching the spec seed exactly.
- Roadmap is implicit in the build order (8 layers) but there is no timeline with dates. CHARTER.md adds milestones (M1-M5) and a phase table mapping layers to review gates.

**Forge SDLC (forge-project-tracker):**
- HANDOFF.md lists deferred items (D01-D27) with version tags (V2, V3, V4+, post-V1) matching the spec.
- Build order has 5 layers instead of 8. No milestones with names. No timeline.
- `project.yaml` captures metadata (status, track, complexity) but no version roadmap.

**Verdict:** Both account for all four version scopes. PreFlight is slightly more structured in its deferred-capabilities organization, and CHARTER.md adds named milestones. SDLC uses numbered deferred items (D01-D27) which are more trackable. Neither has calendar dates, which is appropriate given the spec doesn't define them.

## 3. Build Phases

**Spec Seed:** Defines no build phases (that is the job of the planning outputs).

**PreFlight Build Order (8 layers):**
1. Project Scaffold & Auth
2. Foundation Integration Layer
3. Search & Discovery
4. Document Upload & Classification
5. Generation Engine
6. Day-1 Outputs
7. Admin & Configuration
8. Notifications & Polish

Each layer includes: what to build, dependencies, what it produces, and `<!-- llm-route -->` tags for agent routing.

**SDLC Build Order (5 layers):**
1. Foundation Integration & Project Skeleton (= PreFlight Layers 1+2)
2. Core Search & Document Management (= PreFlight Layers 3+4)
3. Content Generation Pipeline (= PreFlight Layer 5 + part of 6)
4. Engagement Prep, Meeting Notes & Cross-Sell (= PreFlight rest of Layer 6)
5. Admin, Config & Polish (= PreFlight Layers 7+8)

Each layer has a dedicated phase-context.md file with: phase goal, what to build, requirements covered (R-numbers), key constraints, tech stack, dependencies, and research context.

**Mapping to each other:**

| PreFlight Layer | SDLC Phase | Alignment |
|-----------------|------------|-----------|
| 1: Scaffold & Auth | Phase 1 (combined with Layer 2) | Aligned |
| 2: Foundation Integration | Phase 1 (combined with Layer 1) | Aligned |
| 3: Search & Discovery | Phase 2 (combined with Layer 4) | Aligned |
| 4: Document Upload | Phase 2 (combined with Layer 3) | Aligned |
| 5: Generation Engine | Phase 3 | Aligned |
| 6: Day-1 Outputs | Phase 3 (Account Brief) + Phase 4 (rest) | Split differently |
| 7: Admin & Config | Phase 5 | Aligned |
| 8: Notifications & Polish | Phase 5 | Aligned |

**Verdict:** The SDLC consolidates PreFlight's 8 layers into 5 phases, but the work is the same. The SDLC approach is arguably more practical (fewer, larger phases reduce coordination overhead). PreFlight's 8-layer approach is more granular, which is better for tracking but may create unnecessary sequencing. The SDLC's per-phase context files are a significant advantage -- each phase has a self-contained document a builder can read without context-switching.

## 4. Success Criteria / Test Coverage

**PreFlight:**
- HANDOFF.md: 26 success criteria checkboxes, all behavioral and testable ("User can log in via AD/SSO", "User can generate an Account Brief in under 60 seconds with data completeness indicator", "Demo runs reliably 3 consecutive times end-to-end").
- Test Expectations table: unit (pytest + vitest), integration (against live Foundation), smoke (3 manual walkthroughs). Lists specific test targets.
- Definition of Done: 7 items.
- CHARTER.md: 10 success metrics with targets, measurement methods, and notes. Includes quantitative targets (< 60s, < 3s, < 2s, 80%+ reduction).
- PROJECT.md: 28 checkboxes (overlaps with HANDOFF).

**Forge SDLC:**
- HANDOFF.md: 11 success criteria (narrative, not checkboxes). Behavioral and testable ("Account Brief generates in <60 seconds", "No Foundation-permissioned document surfaced to unauthorized user").
- Test Expectations: unit (80%+ coverage target stated), integration (against live Foundation), E2E (3 critical flows named), security tests (MSA opt-out, permissions, GDPR).
- Definition of Done: 6 items.
- PROJECT.md: 10 done-when checkboxes.
- Phase 5 context: 6 explicit E2E test scenarios named with specific accounts.

**Verdict:** PreFlight has more success criteria (26 vs 11 in the HANDOFF), and CHARTER.md adds quantitative metrics with measurement methods. However, the SDLC's test expectations are more specific about security testing (MSA opt-out, permission inheritance, GDPR purge as named test scenarios). The SDLC also explicitly states 80%+ unit test coverage target, which PreFlight doesn't. SDLC Phase 5 context names exact test flows against specific demo accounts.

Overall: PreFlight wins on breadth of success criteria. SDLC wins on specificity of test plan. Both are testable and behavioral (not vague).

## 5. Risk Coverage

**PreFlight (risks-and-considerations.md):**
- 17 risks across 4 categories:
  - Technical: 6 (T1-T6: Foundation API load, Weaviate performance, LLM quality variance, SP write-back permissions, AD/SSO complexity, concurrent generation conflicts)
  - Content/Data: 6 (C1-C6: offerings catalog, taxonomy, demo data, entity resolution, templates, no seed data)
  - Organizational: 4 (O1-O4: adoption, template collection, executive alignment, content owner bandwidth)
  - Scope: 3 (S1-S3: feature creep, degradation testing, admin surface size)
- Each risk has: severity, likelihood, description, impact, mitigation.
- Includes 3 response plans (offerings missing, Foundation unreliable, content dependencies missing).

**Forge SDLC (risks-and-considerations.md):**
- 15 risks across 3 severity tiers:
  - Critical: 1 (RISK-01: offerings catalog)
  - High: 6 (RISK-02 through RISK-07: Foundation availability, taxonomy, LLM quality, SP permissions, MSA opt-out, GDPR cascade)
  - Medium: 5 (RISK-08 through RISK-12: demo data, concurrent generation, Weaviate traversal, LLM rate limits, admin config)
  - Low: 3 (RISK-13 through RISK-15: SP write-back scope, Foundation v4 creep, deployment drift)
- Each risk has: severity, description, evidence (quoted from spec), mitigation, blast radius, reversibility.
- HANDOFF risk summary table: 12 risks with phase-linked mitigation status.

**Risks unique to PreFlight:**
- O1: Adoption beyond EA team
- O2: Template ground truth collection stalls
- O3: Executive alignment not yet obtained
- O4: Content owner bandwidth
- S1: Feature creep from user modes
- S2: Graceful degradation testing complexity
- S3: Admin config surface too large
- T5: AD/SSO integration complexity

**Risks unique to SDLC:**
- RISK-05: SP permission enforcement gaps (PreFlight mentions this in constraints but doesn't call it a standalone risk)
- RISK-06: MSA opt-out enforcement gap (PreFlight mentions opt-out but doesn't escalate to a compliance risk)
- RISK-07: GDPR annotation cascade (PreFlight mentions in constraints, not as a named risk)
- RISK-13: SP write-back permission scope
- RISK-14: Foundation v4 feature creep
- RISK-15: Deployment drift (Contabo to Azure)

**Verdict:** PreFlight covers more risks (17 vs 15) and uniquely surfaces organizational and scope risks that the SDLC misses entirely (executive alignment, content owner bandwidth, feature creep, degradation testing complexity). The SDLC uniquely surfaces compliance risks as standalone items (MSA opt-out, GDPR cascade, SP permissions) which PreFlight buries in constraints. The SDLC risk format is stronger: it includes "evidence" quotes from the spec seed, "blast radius," and "reversibility" -- more useful for decision-making. PreFlight's response plans are a valuable addition the SDLC lacks.

### Bottom Line

The **Forge SDLC output is more useful for building KIRT**. Its phase-context files, numbered requirements, and per-phase tech stack guidance mean a developer (or agent) can pick up any phase and start immediately. The synthesis.md provides research-to-requirements traceability that PreFlight lacks entirely.

The **PreFlight output is more useful for aligning stakeholders**. CHARTER.md with its RACI matrix, named stakeholders, milestones, budget section, and 10 explicit non-goals is the kind of document that keeps a project on track organizationally. The richer personas are better for UX design conversations.

**Optimal approach:** Use both. The SDLC phase-context files as the build guide, the PreFlight CHARTER.md for organizational governance, and merge the risk sets (PreFlight's organizational risks + SDLC's compliance risks).
