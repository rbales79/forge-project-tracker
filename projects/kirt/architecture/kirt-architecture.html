Overview

KIRT (Knowledge In Real Time) is the AI-powered intelligence layer for the Strategic Account Framework (SAF). It sits on top of Forge Foundation (v3.0) and automates the manual work of account research, engagement preparation, and deliverable generation.

KIRT is not the framework. The SAF defines the process -- KIRT makes it faster. Someone can run the SAF manually with CRM data, web research, and templates. KIRT replaces hours of manual work with minutes of AI-assisted generation.

Two-Product Architecture

Foundation -- Shared Data Platform

Ingests, normalizes, and serves data. Admin-only, no user surface. Currently v3.0 on Contabo (Forge ecosystem). Target: Azure enterprise.

KIRT -- Intelligence Layer

User-facing intelligence layer. Greenfield build. All business logic, classification, scoring, generation, and recommendation happens here.

SAF Framework

KIRT automates the Strategic Account Framework's 6 phases. Each phase has defined inputs, outputs, and manual effort that KIRT replaces.

v

  2. Key contacts -- names, titles, roles in buying process, relationship status
  3. Business priorities -- from earnings, 10-K, conversations
  4. Relationship history -- engagement timeline, past deals, current opportunities
  5. Open opportunities -- pipeline with stage, value, next steps, risks

  2. Known pain points -- from previous interactions
  3. Stakeholder map -- pre-populated, user-editable

  5. Discovery questions -- pulled from 200+ question taxonomy, filtered by business/industry/solution modules and mapped to relevant offerings
  6. Next-best-question suggestions -- based on account context

Document Generation Lifecycle

User selects account / opportunity
  --> KIRT pulls applicable template from SP (via Foundation's template canonical)
  --> KIRT triggers research run (pulls SF data + Foundation knowledge + public data)
  --> KIRT generates document using LLM
  --> KIRT writes deliverable to SP (designated output folder)
  --> Foundation records SP URL in derived_docs[] on the research canonical
  --> User sees finished document with provenance summary
  --> User can edit inline or regenerate sections with new instructions

Scoring Models

Three scoring models drive SAF phases. All need at least v0.1 definitions before building the features that consume them.

Growth Scoring Phase 1

Purpose: Account-level score driving tier assignment and prioritization.

Inputs: Revenue, strategic value, relationship depth, engagement frequency, white space size.

Output: Tier assignment (Tier 1/2/3) determining which accounts get full treatment.

V1 Approach: Simplified weighted formula, tunable via admin settings.

Opportunity Scoring Phase 3

Purpose: Signal-based scoring from meeting transcripts and engagement data.

Inputs: Meeting signals, engagement recency, competitive mentions, stakeholder sentiment, deal stage progression.

Output: Opportunity score with human confirmation step.

V1 Approach: Pre-computed at ingest time, stored in Foundation annotations.

Offering Maturity Phase 3

Purpose: Gates whether an offering is ready to propose to a specific account.

Inputs: Offering readiness, delivery track record, available resources, market fit.

Output: Pass/fail gate before Phase 4 deliverable generation.

V1 Approach: Manual rating in the offerings catalog, surfaced in KIRT.

Offerings Catalog Dependency

Biggest Content Blocker The offerings catalog is foundational. White space grid, scoring, cross-sell, and discovery questions all depend on it. It does not exist yet -- this is the single biggest content dependency. The catalog is a content asset, not a KIRT feature. KIRT consumes it. Building and maintaining it is an operational responsibility (admin/settings, configurable per deployment).

The catalog must define:

  • Solution domains and individual offerings within each domain
  • Maturity rating per offering (ready to sell, emerging, sunset)
  • Mapping to discovery question modules (which questions surface which offerings)
  • Versioning (offerings evolve; catalog needs point-in-time snapshots)

Salesforce Relationship

KIRT complements Salesforce -- it does not replace it. SF stays as system of record for structured CRM data. KIRT handles unstructured analysis, semantic search, AI generation, and pattern intelligence.

Stays in Salesforce

  • Account records, pipeline, activity history, forecasting
  • Contact management, standard reporting
  • Workflow automation on structured data

KIRT Adds

  • Unstructured data analysis, semantic search
  • AI-generated deliverables, pattern recommendations
  • Cross-sell engine, tribal knowledge capture

KIRT reads from SF (CSV exports for V1, direct API for V4+). KIRT enriches what SF has. KIRT generates deliverables that SF cannot. If/when a single normalized SF tenant exists, KIRT's value increases -- clean data means better intelligence.

Design Constraints

Nine key design constraints govern KIRT's architecture and behavior.

1

Foundation owns all document storage.

Users upload files through KIRT, which routes them to Foundation via POST /v1/ingest/. KIRT does not maintain its own document store — Foundation is the single storage layer.

2

SP files are never modified by Foundation.

Enrichments live in Foundation only. KIRT writes new files to SP (deliverables); it does not edit source files unless the user explicitly initiates a write-back to a specific canonical source_ref.

3

Dedup is invisible to the user by default.

1 result per logical document. The N-copies detail is accessible but not the default view.

4

Users never see the research process, only the output.

Iceberg snapshots are internal audit/versioning infrastructure, not user-facing content.

5

Permission inheritance is strict.

KIRT never surfaces a document the user could not access in its source system. Enforced pre-query in Foundation, not post-fetch in KIRT.

6

Archival is normalization, not deletion.

Archived SP files remain untouched. Foundation stops treating them as live canonical sources. Stale source alerts if an archived file gets edited.

7

Write-back to SP is explicit and user-initiated.

Every write to SP (deliverable generation, canonical update) is a deliberate user action with a confirmation step.

8

KIRT complements Salesforce, never replaces it.

SF stays as system of record for structured CRM data. KIRT handles unstructured analysis, semantic search, AI generation, and pattern intelligence.

9

Admin-configurable deployment settings.

Directory structures, SP output paths, scoring model weights, offerings catalog, and data source configurations must be tunable per deployment without code changes.
