## What KIRT Is

KIRT (Knowledge In Real Time) is the AI-powered intelligence layer for account research, engagement preparation, and deliverable generation. It sits on top of Forge Foundation (v3.0) and serves anyone who needs account intelligence — from Enterprise Architects following the full Strategic Account Framework to AEs who just need to prep for a meeting in 10 minutes.

KIRT is not the SAF. The SAF is one workflow that runs on KIRT. KIRT's capabilities are organized around the SAF's 6 phases, but every feature is independently accessible. Users don't follow a linear pipeline — they use what they need, when they need it.

**Two-product architecture:**
- **Foundation** — shared data platform. Ingests, normalizes, serves. Admin-only, no user surface. Currently v3.0 on Contabo (Forge ecosystem). Target: Azure enterprise.
- **KIRT** — user-facing intelligence layer. Greenfield build. All business logic, classification, scoring, generation, and recommendation happens here.

**Source-of-truth model:**
- **SharePoint** = document source-of-truth (files live here)
- **Foundation** = raw data source-of-truth (ingested, normalized, searchable)
- **KIRT** = intelligence source-of-truth (enrichments, classifications, scores, generated insights)

Research, scoring, and intermediate analysis live in the data lake only. Final deliverables write to SP. End users see the deliverable in SP and the full context in KIRT. "Baby, not the baby-making."

## User Modes

KIRT serves three distinct usage patterns. The build must support all three from V1.

| Mode | Who | How They Use KIRT |
|------|-----|-------------------|
| **Full SAF** | Enterprise Architects, strategic account leads | Sequential phases, cumulative intelligence, scoring pipeline, all deliverable types. SAF progress tracking per account. |
| **On-Demand Intelligence** | AEs, Solution Specialists, individual contributors | Ask a question, get an answer. Generate briefs, prep for meetings, upload notes, search. No framework required. This is the Nighthawk replacement. |
| **Consumer** | Sales Leadership, Delivery/PM, extended team | Search, browse, read. Consume intelligence generated by others. No generation, no upload. |

**Design principle:** The UI is task-oriented ("What do you need?") not phase-oriented ("Which SAF step are you on?"). SAF progress tracking exists as an optional overlay for EA users — non-EA users never see it unless they opt in.

## SAF Framework Mapping

The SAF's 6 phases organize KIRT's capabilities. They describe **what KIRT can do**, not what users must do. Every feature is independently accessible — users can generate an Engagement Prep without completing Account Intelligence first, and EAs can skip phases depending on where they are in a relationship cycle.

| Phase | What It Does | Manual Effort | What KIRT Does | Requires SAF? |
|-------|-------------|---------------|----------------|--------------|
| 1. Account Intelligence | Growth scoring, white space grid, public data | 2-4 hrs/account | Auto-generates Account Briefs, monitors public signals, maintains live white space grid, surfaces cross-sell recommendations | Brief: No. Scoring/white space: Yes. |
| 2. Prepared Engagement | Meeting brief, discovery questions, stakeholder map | 1-2 hrs/meeting | Generates Engagement Prep packages, suggests next-best-questions, pre-populates stakeholder maps | No — works with whatever context exists |
| 3. Discovery & Scoring | Signal capture from transcripts, opportunity scoring, maturity gate | 30-60 min/meeting | Analyzes transcripts for signals, auto-scores opportunities (human-confirmed), routes intelligence | Upload/extract: No. Scoring: Yes. |
| 4. Client Deliverables | Reframe deck, assessment kickoff, roadmap, EBR/QBR | 2-8 hrs/deliverable | Drafts all formats at 80-90% quality; human refines via inline editing and section regeneration | Basic generation: No. Gated generation: Yes. |
| 5. Enablement & Progression | Insight drops, win/loss patterns, advancement triggers | Institutional memory | Tracks outcomes, generates QBR/EBR reports, surfaces patterns, monitors triggers | Yes |
| 6. Continuous Threads | Stakeholder tracking, outcome register, white space, governance | Spreadsheets & tribal knowledge | Centralized, always-current, role-based access to all living artifacts | Yes |

**Target metrics:** 80%+ reduction in document preparation and engagement prep time. 50-70% improvement in account intelligence quality.

## Graceful Degradation

Every feature must work with partial data and clearly communicate what's missing. KIRT does the best it can with what it has and tells the user what's missing.

**Data completeness indicators** on every generated output:
- Show which data sources were available (SF record, SP documents, meeting notes, public data, stakeholder map, offerings catalog)
- Flag what's missing and how it affects output quality
- Suggest specific actions to improve ("upload a stakeholder map", "complete discovery session")
- Display a quality indicator (e.g., 75% — based on data coverage, not content quality)

**Degradation rules:**
- Account Brief without SF data → still generates from SP docs + public data, flags SF gap
- Engagement Prep without prior meetings → generates questions and stakeholder placeholders from account context alone
- Cross-sell without offerings catalog → pattern-based suggestions from engagement history ("accounts like this also engaged with security assessments") — less precise but still useful
- Discovery questions without formal taxonomy → AI-generates contextual questions from account data. The taxonomy improves precision but isn't required.
- Any output with no data at all → tells the user what to upload or connect, doesn't generate empty content

## Core User Flows

### 1. Search & Discovery

- User searches across documents, research, account data
- Foundation returns 1 canonical result per logical document (dedup filter applied before results reach KIRT)
- KIRT surfaces: title, doc_type, summary, last modified, source system, "N copies found" indicator if `source_refs[]` has >1 entry
- Clicking a result opens the canonical view — not a raw file link
- User can expand to see all source locations if needed (advanced / power user)

**Search must support:**
- Keyword/phrase (BM25)
- Semantic (vector)
- Relationship traversal via Weaviate cross-references ("show me all MSAs related to this client")
- Structured filters: doc_type, date range, client, author, status
- All modes blend in a single query — not separate search tabs

**Search result ranking (knowledge hierarchy):**
1. Account intelligence dossier / brief for the specific account
2. Recent meeting notes and transcripts (last 90 days)
3. Salesforce data (account, opportunity, contact records)
4. Public research (SEC filings, earnings, news)
5. Playbooks, battle cards, and templates

**Data freshness:** Search results and account intelligence surfaces must indicate data age. Flag anything >30 days as potentially stale.

### 2. Duplicate / Variant Management

- KIRT shows 1 document by default
- "N copies found" badge visible on the result card
- Expanding reveals: all source locations, `is_canonical` marker, last modified per location
- **V1:** Exact duplicate detection via content hash. Default canonical selection via admin-configurable location-ranking (ranked list of source locations by authority). User can manually override.
- **V2:** Near-duplicate detection (fuzzy matching). KIRT surfaces a "reconcile" prompt: "3 variants of this document exist. View differences?"
- User can set a source_ref as canonical (writes `is_canonical` flag back to Foundation)
- User can mark non-canonical copies as archived (Foundation updates `status: archived` — does not delete SP files)
- **KIRT does not decide which copy is authoritative — the user does, with KIRT surfacing the data needed to decide**

### 3. Source Management & Ingestion

**Connecting new data sources:**
- Admins can initiate ingestion of new SP sites, document libraries, and data sources directly through KIRT's admin/config UI
- KIRT triggers Foundation's connector infrastructure (`POST /v1/ingest/`, `GET /v1/connections/`) — source configuration is a KIRT admin function, not a Foundation-only task
- Source connections are admin-configurable: which SP sites to crawl, which SF exports to watch, which folders to monitor
- Output destinations (where KIRT writes deliverables back to SP) are also configured here — separating input sources from output targets

**User uploads:**
- Users can upload files directly through KIRT (meeting notes, transcripts, ad-hoc documents, research)
- Uploaded files are ingested into Foundation via the standard `POST /v1/ingest/` pipeline — KIRT does not maintain a separate document store
- On ingest, KIRT auto-classifies documents by type:

| Classification | Processing Route |
|---------------|-----------------|
| Meeting notes | NLP extraction: signals, entities, action items → Phase 3 signal capture |
| Transcript | Speaker diarization, signal extraction, sentiment analysis |
| BSE / account profile | Entity extraction, account linking, white space input |
| SOW / contract | Offerings linkage, engagement tracking, maturity input |
| Deck / presentation | Subtype classification (pitch vs. deliverable vs. internal), theme extraction |
| Template | Mark as template, feed generation pipeline, never modify original |
| KIRT-generated | Provenance tracking, version control, link to source inputs |

- Classification happens at ingest, LLM-based with labeled training examples
- Misclassification: manual override by user (v1), feedback loop to retrain (v2+)

### 4. Document Generation Lifecycle

```
User selects account / opportunity
  → KIRT pulls applicable template from SP (via Foundation's template canonical)
  → KIRT triggers research run (pulls SF data + Foundation knowledge + public data)
  → KIRT generates document using LLM
  → KIRT writes deliverable to SP (designated output folder)
  → Foundation records SP URL in derived_docs[] on the research canonical
  → User sees finished document with provenance summary
  → User can edit inline or regenerate sections with new instructions
```

**What the user sees:** the output document link (in SP) + provenance summary ("Generated from [template name], [date], research run #N"). They do not see intermediate enrichment, raw research data, or the generation process.

**What lives where:**
- Research run data → Iceberg (versioned, auditable, internal only)
- Output document → SharePoint (user-accessible, final deliverable)
- Lineage + provenance → Foundation canonical record (surfaced on demand in KIRT)

**SP output conventions:**
- Deliverables write to configurable output folders per account/engagement
- File naming: `[Client]-[DocType]-[Date].docx` (e.g., `Acme-Corp-Account-Brief-2026-Q1.docx`)
- Shallow folder structures for discoverability (flat over deep)
- Generated docs favor prose + structured sections over raw tables (better for downstream search and Copilot consumption)

Re-running research creates a new Iceberg snapshot on the same research record and generates a new SP document revision. Previous versions remain in SP version history.

**Content versioning:** Generation metadata (timestamp, input hash, generation count) stored as Foundation annotations. SP handles actual document versions. KIRT shows "Generated v3, last updated 2026-03-20" as a badge — no full diff view in V1.

**Concurrent generation:** If two users generate deliverables for the same account simultaneously, both complete. Both write to SP (different filenames or SP versions). KIRT surfaces both with a "2 versions generated" indicator. User picks which to keep. Content merging is V3+ (requires semantic diff).

**Feedback learning:**
- V1: Prompt-level feedback — store edited outputs alongside originals. When generating for the same account/type, include user's prior edits as few-shot examples in the prompt. No model training.
- V2: RAG feedback loop — user edits ingested back into Foundation as "KIRT-refined" documents. Future generations retrieve the refined version as context.
- V3+: Fine-tuning evaluation — collect edit pairs (generated vs. user-corrected) as training data. Fine-tune if edit volume justifies it.

**Notifications:**
- V1: In-app badges and dashboard indicators (new data available, brief is stale, generation complete). Comes free with the React UI.
- V2: Email digest, push notifications, configurable per user. Requires notification service and user preference management.

**Resilience:** V1 shows an error state when Foundation is unavailable. Cached/degraded mode is a V2+ concern. For demo reliability, ensure Foundation stability during demo windows rather than engineering around it.

### 5. Permissions & Access Control

**Document-level permissions (Foundation-enforced):**
- KIRT authenticates users via their M365/Entra identity
- JWT claims passed to Foundation query API on every request
- Foundation enforces: user's M365 group memberships must intersect with `sp_permissions[]` on the canonical record
- If a user cannot see a file in SharePoint, they cannot see it in KIRT — no exceptions

**KIRT role-based access:**
- V1: Single AD group with full access. All authenticated users get the complete platform.
- V2+: Bronze/silver/gold tiers via AD-group-to-role mapping, plus per-account filtering once SF ownership data is clean.

| Tier | Personas | Access | Version |
|------|----------|--------|---------|
| Full (V1) | All authenticated users | Complete platform — all features, all accounts | V1 |
| Gold | Architect/EA, Sales Leadership | Full platform — admin settings, scoring config, analytics | V2+ |
| Silver | AE, Solution Specialist | Search, briefs, engagement prep, cross-sell for accessible accounts | V2+ |
| Bronze | Delivery/PM | Read-only briefs, engagement context, knowledge export | V2+ |

**MSA opt-out handling:**
- Opt-out means keyword search only — documents are ingested and indexed for BM25 keyword/phrase search, but no LLM processing (no summarization, no AI classification, no generation using that content)
- Requires `ai_processing_allowed` flag on canonical records; KIRT checks this before running any LLM pipeline
- Opt-out content is excluded from generation context

## Day-1 Outputs (MVP Deliverables)

These are the five outputs KIRT must generate for the MVP demo:

### Account Brief
1-2 page summary generated in <60 seconds. Sections:
- Company overview (industry, size, revenue, headquarters)
- Key contacts (names, titles, roles in buying process, relationship status)
- Business priorities (from earnings, 10-K, conversations)
- Relationship history (engagement timeline, past deals, current opportunities)
- Open opportunities (pipeline with stage, value, next steps, risks)
- Competitive landscape (who else, strengths/weaknesses)
- Strategic notes (personal observations, dynamics, sentiment)
- White space analysis (what we sell vs. what this account buys)
- Action items

### Engagement Prep
Pre-meeting package:
- Prior work with this account
- Known pain points from previous interactions
- Stakeholder map (pre-populated, user-editable)
- Recommended talking points
- Discovery questions (pulled from the 200+ question taxonomy, filtered by business/industry/solution modules and mapped to relevant offerings)
- Next-best-question suggestions based on account context

### Meeting Summary & Follow-Up
The on-demand pipeline: user uploads meeting notes → KIRT immediately generates a client-facing output. No account context required — works from the notes alone, enriched by whatever account data exists.

Generates:
- Polished meeting recap (what was discussed, key takeaways, decisions made)
- Structured action items with owners and dates (extracted from notes)
- Draft follow-up email/summary suitable for sending to the client
- Account intelligence update (what changed — new pain points, competitive mentions, stakeholder signals fed back into account context)

This is the lowest-friction entry point for on-demand users. Upload notes, get something sendable. Single action, immediate value.

### Client Deliverable
AE walks in, inputs company name + a few keywords, generates client-ready documentation with account history. The canonical adoption use case — lowest friction, highest perceived value.

### Cross-Sell Recommendations
Pattern-based suggestions derived from historical engagement data:
- "Accounts like this also bought..." using engagement patterns + unstructured signals
- Triggered after data ingestion or meeting note upload
- Requires the offerings catalog as the reference frame

## Scoring Models

Three scoring models drive SAF phases. All need at least v0.1 definitions before building the features that consume them.

### Growth Scoring (Phase 1)
- Account-level score driving tier assignment and prioritization
- **Inputs:** Revenue, strategic value, relationship depth, engagement frequency, white space size (composite — no single dimension is sufficient)
- **Output:** Tier assignment (Tier 1/2/3) determining which accounts get full treatment
- **V1:** Simple weighted formula starting with equal weights across dimensions, tunable via admin settings, with manual override. Users can adjust tier assignments when the formula gets it wrong. Precision is less critical at launch because of manual override.
- **V2+:** ML-informed scoring once data volume justifies it

### Opportunity Scoring (Phase 3)
- Signal-based scoring from meeting transcripts and engagement data
- **Inputs (to define):** Meeting signals, engagement recency, competitive mentions, stakeholder sentiment, deal stage progression
- **Output:** Opportunity score with human confirmation step
- **V1:** Pre-computed at ingest time, stored in Foundation annotations

### Offering Maturity Scoring (Phase 3)
- Gates whether an offering is ready to propose to a specific account
- **Inputs (to define):** Offering readiness, delivery track record, available resources, market fit
- **Output:** Pass/fail gate before Phase 4 deliverable generation
- **V1:** Manual rating in the offerings catalog, surfaced in KIRT

## Offerings Catalog

The offerings catalog is foundational. White space grid, scoring, cross-sell, and discovery questions all depend on it. **It does not exist yet — this is the single biggest content dependency.**

**V0.1 approach:** Best-effort extraction from existing materials — PPT decks and practice collateral are the likely source. Extract as close to what we actually have today rather than building from scratch or deferring with a placeholder.

The catalog must define:
- Solution domains and individual offerings within each domain
- Maturity rating per offering (ready to sell, emerging, sunset)
- Mapping to discovery question modules (which questions surface which offerings)
- Versioning (offerings evolve; catalog needs point-in-time snapshots)

The catalog is a content asset, not a KIRT feature. KIRT consumes it. Building/maintaining it is an operational responsibility (admin/settings, configurable per deployment).

## Data Normalization Strategy

KIRT's data flow follows a clear directional pattern: sources are consumed, intelligence is generated, outputs go to a new location, and originals are managed in place.

```
Source SP sites/SF/uploads → Foundation ingests and normalizes → KIRT enriches and generates
  → Generated deliverables write to designated SP output location (never back to source)
  → Source documents marked stale when superseded by KIRT-generated versions
  → Archived copies stop appearing in default search results
```

**Input vs. output separation:**
- **Input sources** (configured via KIRT admin): SP document libraries, SF CSV exports, user uploads, public data APIs. Foundation crawls these.
- **Output targets** (configured via KIRT admin): Designated SP folders where KIRT writes generated deliverables. Always separate from input sources.
- This separation is the normalization strategy — source documents live where they were created, KIRT outputs live in a clean, structured location. No mixing.

**Lifecycle:**
- Source documents are ingested and indexed but never modified by Foundation or KIRT
- When KIRT generates a deliverable that supersedes a source document, the source can be marked as archived (user-initiated, per design constraint #6)
- Archived documents remain in SP but stop appearing as active canonical sources in Foundation
- If someone edits an archived SP file, Foundation detects the change and surfaces a "stale source" alert in KIRT
- The output location becomes the new authoritative location for KIRT-generated content

**Admin configuration surface (KIRT config):**
- Add/remove SP sites and document libraries as input sources
- Configure output SP folders per account, engagement, or deliverable type
- Set crawl schedules and delta sync frequency
- Define location-ranking for canonical selection
- All source and output configuration is admin-managed, not hard-coded

## Key Design Constraints

1. **Foundation owns all document storage.** Users upload files through KIRT, which routes them to Foundation via `POST /v1/ingest/`. KIRT does not maintain its own document store — Foundation is the single storage layer.

2. **SP files are never modified by Foundation.** Enrichments live in Foundation only. KIRT writes new files to SP (deliverables); it does not edit source files unless the user explicitly initiates a write-back to a specific canonical source_ref.

3. **Dedup is invisible to the user by default.** 1 result per logical document. The N-copies detail is accessible but not the default view.

4. **Users never see the research process, only the output.** Iceberg snapshots are internal audit/versioning infrastructure, not user-facing content.

5. **Permission inheritance is strict.** KIRT never surfaces a document the user couldn't access in its source system. Enforced pre-query in Foundation, not post-fetch in KIRT.

6. **Archival is normalization, not deletion.** Archived SP files remain untouched. Foundation stops treating them as live canonical sources. Stale source alerts if an archived file gets edited.

7. **Write-back to SP is explicit and user-initiated.** Every write to SP (deliverable generation, canonical update) is a deliberate user action with a confirmation step.

8. **KIRT complements Salesforce, never replaces it.** SF stays as system of record for structured CRM data. KIRT handles unstructured analysis, semantic search, AI generation, and pattern intelligence.

9. **Admin-configurable deployment settings.** Directory structures, SP output paths, scoring model weights, offerings catalog, LLM provider selection, and data source configurations must be tunable per deployment without code changes.

10. **Single-tenant for V1.** Internal Pellera deployment only. Multi-tenancy is a future version concern gated by security and legal review. Foundation's `tenant_id` scoping means the data layer supports multi-tenancy — KIRT can add it when the business requires it without a rewrite.

## Data Retention & GDPR Purge

KIRT manages its own purge workflow. Foundation provides the mechanism (`DELETE /v1/delete/<id>/` with GDPR hard purge option), but triggering is KIRT's responsibility — KIRT owns the business logic for what to purge and when.

- **V1:** Admin-triggered purge actions.
- **V2+:** Automated retention policies.

**Annotation cleanup is KIRT's responsibility.** Foundation's purge cascade only covers documents and chunk records — it does not cascade to annotations. When a source document is purged, KIRT must explicitly delete its own annotations (classifications, scores, tags) via the Foundation annotations API.

KIRT MVP builds on Foundation v3.0. No v4 features are required.

**V3 capabilities KIRT depends on:**

**V4 capabilities that would help later versions:**
- Automated reconciliation → useful for V2 near-duplicate management at scale

**Recommendation:** Build KIRT V1 on Foundation v3 now. Plan Foundation v4 work in parallel with KIRT V2 planning so v4 capabilities land before KIRT needs them at scale.

**Foundation is running.** v3.0 is deployed and operational on Contabo. No seed data exists yet — all testing should be done against the live instance when possible.

## Development & Testing Strategy

**Local first, Foundation last.**

KIRT development should be testable locally without Foundation wherever possible (mock API responses, local test data). But integration testing must run against the live Foundation v3 instance to catch real-world issues.

- **Unit tests and UI development:** Local. Mock Foundation API responses. Fast iteration.
- **Integration tests:** Against live Foundation v3. Tests the real API contract, real Weaviate search behavior, real ingestion pipeline.
- **Foundation flaws:** If testing against Foundation reveals a bug or missing capability, create an issue in the Foundation repo. Don't work around it in KIRT — fix it at the source.
- **No seed data in Foundation today.** Demo data (per `demo-data-requirements.md`) needs to be ingested into Foundation before end-to-end demo testing.
- **Test both directions:** KIRT bugs are KIRT issues. Foundation API issues, search quality issues, ingestion failures are Foundation issues. File them separately and track independently.

## Version Scope

### V1 — MVP (SAF Phases 1-2 capabilities, all three user modes)

**Goal:** Credible demo of account intelligence and engagement prep for both SAF users and on-demand users. Non-technical stakeholder watches it and understands the value without explanation. Runs reliably on 3 consecutive attempts.

**Delivers:**
- Task-oriented UI ("What do you need?" — search, prep for meeting, generate brief, upload documents)
- All three user modes supported (Full SAF, On-Demand Intelligence, Consumer)
- Data completeness indicators on every generated output (sources used, gaps, quality score, improvement suggestions)
- Graceful degradation — every feature works with partial data, clearly communicates what's missing
- SSO login (live AD/SSO) with single AD group full access (V1)
- Hybrid search (keyword + semantic, deduplicated results, freshness indicators)
- Account Brief generation (<60 seconds) — works with whatever data exists, doesn't require Phase 1 completion
- Engagement Prep generation (meeting brief + discovery questions + stakeholder map) — works without prior Phase 1
- Meeting Summary & Follow-Up (upload notes → polished recap + action items + draft follow-up email)
- AI-generated discovery questions from account context (formal taxonomy improves precision but isn't required)
- Cross-sell recommendations (pattern-based from engagement history — offerings catalog makes it more precise but isn't required)
- Document upload → Foundation ingest → auto-classification
- Meeting note upload → signal extraction → account enrichment
- Exact duplicate detection (content hash), 1-result UX with "N copies" badge
- Human-in-the-loop editing (inline edit, section regeneration)
- Prompt-level feedback learning (store edits, use as few-shot context for future generation)
- Deliverable write-back to SP output folders with generation metadata annotations
- In-app notification badges and dashboard indicators (new data, stale briefs, generation complete)
- Concurrent generation handling (write both, user picks winner)
- Responsive web UI (mobile read access for search, briefs, engagement prep)
- SAF progress tracking as optional overlay for EA users (not default view)
- Error state when Foundation is unavailable
- Admin-configurable settings (SP paths, directory structures, scoring weights, LLM provider, offerings catalog)
- Health endpoint, error tracking, basic usage logging
